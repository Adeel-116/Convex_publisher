3. UI Components & Layouts

Layouts (ViewGroups)

  1. LinearLayout: Arranges elements sequentially
    • orientation: vertical/horizontal determines arrangement direction
    • weight: distributes space proportionally among children
    <LinearLayout
    android:layout_width="match_parent"
    android:layout_height="wrap_content"
    android:orientation="vertical">
    <!-- Child views arranged vertically -->
</LinearLayout>
  2. RelativeLayout: Positions elements relative to parent or siblings
    • Elements positioned using rules (below, toRightOf, alignParentBottom, etc.)
    • More complex than LinearLayout but flexible for complex UIs
  3. ConstraintLayout: Modern layout with constraint-based positioning
    • Flat view hierarchy (better performance)
    • Responsive to different screen sizes
    • Constraint handles define relationships between views

View Components (UI Elements)

  1. TextView: Display text
    • Non-editable text display
    • Supports formatting, styling, and text appearance customization
  2. EditText: Input field
    • Subclass of TextView that allows editing
    • Different input types (text, number, password, email, etc.)
  3. Button: Clickable button
    • Triggers actions when clicked
    • Can be customized with text, background, style

4. View Binding & Event Handling

View Binding (Connecting UI to Code)

Traditional findViewById method

// In onCreate() method
Button submitButton = findViewById(R.id.submit_button);
EditText nameField = findViewById(R.id.name_field);

Event Handling

  • Responding to user interactions through listeners
  • Common events: click, long-click, touch, focus change

Click Listeners (Most Common)

submitButton.setOnClickListener(new View.OnClickListener() {
    @Override
    public void onClick(View v) {
        String name = nameField.getText().toString();
        // Process the input
    }
});

Toast Messages (Simple Feedback)

  • Brief, non-interactive popup messages
  • Duration: LENGTH_SHORT (2 seconds) or LENGTH_LONG (3.5 seconds)
Toast.makeText(MainActivity.this, "Your message here", Toast.LENGTH_SHORT).show();

5. String Resources & Hardcode vs Softcode

Hardcoded Values (Not Recommended)

  • Values directly embedded in code or layout
  • Problems: difficult to update, doesn't support localization

String Resources (Softcoded - Recommended)

  • Values defined in resource files
  • Benefits: centralized management, localization support, easier updates
  • strings.xml: Contains text values
  • dimens.xml: Dimension values
  • colors.xml: Color values

6. Intent & Navigation

Intent Class

  • Messaging object that communicates between components
  • Core functionality:
    • Starting activities
    • Starting services
    • Delivering broadcasts

Types of Intents

  1. Explicit Intent: Targets specific component by name
    • Used for communication within same app
    • Component is specified directly
  2. Implicit Intent: Declares general action
    • System finds compatible component
    • May show chooser if multiple options available

Starting a New Activity

// Explicit Intent
Intent intent = new Intent(MainActivity.this, SecondActivity.class);
startActivity(intent);

Passing Data Between Activities

  • Using extras (key-value pairs) for data transfer

8. ListView & RecyclerView

ListView

  • Traditional component for displaying scrollable lists
  • Less efficient for long lists
  • Uses Adapter pattern to populate data

RecyclerView

  • Modern, more efficient replacement for ListView
  • Only creates views for visible items plus buffer
  • Uses ViewHolder pattern to recycle view objects

Key Components of RecyclerView

  1. RecyclerView: Container widget
  2. Adapter: Provides data and creates ViewHolders
  3. ViewHolder: Holds and recycles views (required)
  4. LayoutManager: Arranges items (Linear, Grid, Staggered Grid)

10. Array & ArrayList

Array

  • Fixed-size, contiguous memory allocation
  • Direct, fast element access via index
  • Cannot change size after creation
  • Can store primitive types or objects

ArrayList

  • Dynamic-size collection (java.util.ArrayList)
  • Part of Java Collections Framework
  • Automatically resizes as elements added/removed
  • Only stores objects (not primitives)

Comparison

  • Array:
    • Faster access (direct memory)
    • Fixed size (must know size in advance)
    • Less memory overhead
    • Supports primitive types directly
  • ArrayList:
    • Dynamic resizing (more flexible)
    • More memory overhead
    • Only stores objects (uses wrappers for primitives)
    • More built-in methods (add, remove, clear, etc.)

11. ViewHolder Pattern

  • Design pattern for efficient list rendering
  • Caches references to views to avoid repeated findViewById() calls
  • Essential for RecyclerView, optional but recommended for ListView

Benefits

  • Significantly improves scrolling performance
  • Reduces memory allocation and garbage collection
  • Standard practice in modern Android development

12. Custom Layout

Creating a custom layout involves designing UI for specific use cases:

Process

  1. Design XML layout file with desired views and structure
  2. Create adapter class (extends BaseAdapter or RecyclerView.Adapter)
  3. Inflate layout in adapter's getView() or onCreateViewHolder()
  4. Bind data to views in adapter

Benefits

  • Tailored UI for specific needs
  • Consistent design across app
  • Better user experience

13. Activity Lifecycle

Lifecycle States and Methods

  1. onCreate(): Activity is created (initialize UI, variables)
  2. onStart(): Activity becoming visible
  3. onResume(): Activity in foreground and interactive
  4. onPause(): Activity losing focus (partially visible)
  5. onStop(): Activity no longer visible
  6. onDestroy(): Activity being destroyed
  7. onRestart(): Activity restarting after being stopped

Lifecycle Scenarios

  • Rotation: destroy and recreate (onCreate -> onDestroy -> onCreate)
  • Back button: finish activity (onPause -> onStop -> onDestroy)
  • Home button: move to background (onPause -> onStop)
  • Another activity starts: pause current (onPause -> onStop)

14. Quick Reference Tips for Exam

Key Components

  • Android Manifest: Central configuration file
    • Required permissions
    • Declared components (Activities, Services, etc.)
    • Intent filters
    • Application metadata
  • Activity Lifecycle Order:
    • Created: onCreate -> onStart -> onResume
    • Destroyed: onPause -> onStop -> onDestroy
  • Best Practices:
    • Use string resources instead of hardcoded strings
    • Design for different screen sizes
    • Follow Material Design guidelines
    • Use meaningful IDs for views
    • Implement ViewHolder for lists
